Spring Security 浅析

发表于 2019-10-23 更新于 2019-12-18 阅读次数：

权限管理，一般指根据系统设置的安全规则或者安全策略，用户可以访问而且只能访问自己被授权的资源，不多不少。权限管理几乎出现在任何系统里面，只要有用户和密码的系统。

理论篇

认证

在微服务中登录功能将会已一个独立的服务跑着。提供各种方式的登录操作。有密码登录，验证码登录，第三方授权登录。在登录成功以后，将返回登录凭证token，在后续的交互中，均需要携带此token。为了保证token的安全，防止因token显露而导致的安全问题。一般token都会有效期，并且要在用户推出登录的时候将token标记为无效。

密码登录

用户需要提供一个可公开的唯一id【邮箱，账号，手机号】+ 一个只有用户自己知道，也就是密码。如果同时存在这三个就需要规定约束一下，包含@的就是邮箱，纯数字的就是手机号，其他就是账号（不能包含@，也不能是纯数字）。不想加约束的话， 就需要在登录界面加一个当选按钮，所以我更加倾向于使用前者。

第三方授权登录

在生活中，经常会遇到需要在授权登录。这种方式登录，有一个好处就是免注册。例如我想在一个网站上创建账号，我们需要先进入注册界面，用户需要输入个人的基本信息，以及上传头像，如果这个网站对于一个用户不是很在意的话，他可能就会因为太麻烦而不去创建用户了。但如果这个网站接入了第三方比如微信，那么用户只需要扫二维码就行，当前位置就可以从微信那边获取到你的个人信息了，例如头像，昵称。这样会不会不安全？其实当前网站所能获取到的用户信息是非常有限的，像微信号，手机号，这种隐私信息是不会返回的，所以大可方向。

接下来来讲讲，到底这种方式怎么实现的。

我们拿微信来举例，到底用户在扫二维码以后方式了事情。先来讲讲这个二维码是怎么生成的，其实这个二维码就是一个地址, http://qq.com/xxxxxx 微信识别出这个是一个http开头的时候，就会默认用微信内置的浏览器打开这个网站。拿问题来了，微信怎么知道你是在哪个网站上扫这个二维码的呢。说明这个地址是微信和网站商量过的，也就是说，网站提前去微信那边注册，注册结果就是获取到一个appId， 也就是 http://qq.com?appId=123456 ，当我们打开这个连接的时候，微信就知道是谁了。 如果仅仅是这样，是不是很不安全。因为任何人只要打开这个地址就会拿到用户的信息。实际情况是，微信放回的是一个随机字符串code，这个和短信验证码一样，有一个有效期，五分钟左右。这样就做到了，即使被公开没事。为什么会没事？我还不是要拿这个code去微信那边换取用户信息。这个时候就引出了第二个字段screct, 这个字段也是微信提供的，需要保存在自己服务器上，当微信返回code，需要将code通过前端网页传递给服务器，服务器拿到这个code，会去访问微信，同时带上screct，以及appid，当 appId，screct， code都是对的时候，微信就会把用户信息回应给服务器，然后服务器在返回给网页。整个基本流程架构就走完了。其实情况要比整个基础架构来的复杂。后面具体业务代码在做分析。

权限 / 菜单 / 接口

权限是干嘛用的呢？想想看，不同部门看到内容是不是要不一样，学校里面老师和学生看到界面也应该不一样吧。想想看，如果一个手机APP，既可以学生登录，也可以老师登录，那是不是得提前把所有界面都在APP里面写好。通过登陆者的类型判断出应该显示的界面。这种情况就可以通过角色去区分，APP开发者只要一个if就可以搞定了。再来说说安全性问题，如果这个时候APP遭到破解，强行把这个if语句改掉，那是不是也可以看到老师才能看到的内容了。显然写服务器代码的人也需要在APP访问服务器的时候，根据token，判断出当前用户类型，也得加一个if。这两个if，侧重点不同，平时起作用的是前面一个，但是后一个才是真正起到防御作用的。权限 / 菜单 / 接口 这三个词就可以解释通了，权限对于前端来说，就是显示的页面不同，也就是菜单也可以按钮等其他组件，对于后端来说，就是对外开放的接口。所以权限要看使用场景，找对应的意思，权限是一个工程中，最复杂的部分了。如果把权限去掉，其实很多工程就是对数据库的增删改查了。

角色

上文中老师和学生只是提前规定，也就不灵活，得提前知道有什么角色。如果以后要加一个学生会角色。由于学生会是是学生构建用来管理学生的，那这个角色部分老师的管理学生的权限，也有学生自己的权限。这个是理想情况，一般都是加新角色，就要加一定量的新的接口。

动态创建角色，就是动态调整权限。一个角色就是对一串权限的集合。无法就是对这个集合中的元素进行动态的添加和修改。这个和上文中的用if就不行了，因为开放APP的人并不能提前知道有哪些角色，也就不能用角色类型去判断了。这个if是没有了，但是服务器的那个if还在，所以其实很简单，只需要让服务器把那个if的实现发给APP开发者就行，让APP开发者也跟着做一遍就好。那问题又来了，这样子有什么意义没。我后端服务都实现了，你前端在走一次。我只能说，是为了方便用户，或者是更加友善。如果前端不走一次，那就意味着，用户在不点击按钮的时候，是不知道自己有没有这个功能的。

部门

角色只是解决了不同用户具有不同的权限，能看到不同的界面，调用不同的接口。拿老师来将，同样是班主任这个角色。我是不是应该只能看到自己班级的学生信息，而不是全校的学生信息。这个角色并没有帮我们解决。如果我们把角色赋予部门的概念。把某班的班主任和那几十个学生都放在这个部门下，所有赋予的角色，只能在当前部门下有效。灵活创建部门又是一个难点，看实践篇再说。

租户

如果这个APP需要服务于不同的学校，然后学生都是使用学号登录的。是不是就遇到了不同学校，学号相同的情况。最暴力的方式，一个学校一套服务，但为了维护性，以及省钱，这个时候，就需要在登录的时候，让用户选择不同的学校。另一种方式就是不同的学校，有不同的APP，我通过把学校的ID写死在APP里面，这种方式有什么好处呢，我可以定制化APP了，但是在网页上就比较常见，会通过不同的域名，来作为登录界面，通过域名来判断不同的租户。xxx.qq.com xxx1.qq.com 这就是两个或者是qq.com/xxx qq.com/xxx1。登录下自己的qq空间就知道了。
